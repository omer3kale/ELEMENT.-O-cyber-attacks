Module 3 · Security Tasks

Security Task Generator

SecurityTaskGenerator is the bridge between the attack catalog and the scheduler. It derives a task title from each attack, optionally stamps it with a brand prefix, and hands the full list to ProcessableItemsService to be distributed across your analysts' calendars.

Add Multi-Tenant → See the Attack Catalog

Architecture

How SecurityTaskGenerator works

The generator is constructed with three collaborators:

  • AttackRepository — provides the full AttackAggregate[] array.
  • ProcessableItemsService — schedules the derived task titles.
  • ?BrandAwareTaskTemplates — optional; enables brand prefixes and per-attack overrides.

When generateAndSchedule() is called, the generator walks every attack in the repository in order, builds a task title string, and assigns it round-robin to the list of User objects. This guarantees that every analyst receives a representative spread of attack categories rather than all phishing tasks going to one person and all cloud attacks to another.

Naming convention

The [CATEGORY][Brand] Task Title convention

Every task name is assembled in three layers. Outer brackets are added left-to-right; the core title comes from the attack's .attack file. 

  Attack name from DSL:  "Phishing Email Campaign"
  Category tag:          [SOCIAL-ENG]
  Brand prefix (opt.):   [ACME Bank]

  Without brand:         [SOCIAL-ENG] Run phishing simulation for Phishing Email Campaign
  With brand:            [ACME Bank] [SOCIAL-ENG] Run phishing simulation for Phishing Email Campaign
  With override:         [ACME Bank] Custom: Train staff on targeted BEC mails
Override takes full priority. If a BrandAwareTaskTemplates instance has an explicit override for an attack name, that string is used verbatim and no category tag or brand prefix is prepended.

Value object

BrandAwareTaskTemplates

An immutable value object that carries a brand name and an optional map of per-attack title overrides. Copy-on-write methods (withBrandName(), withOverride()) return new instances without mutating the original.

Constructing and using BrandAwareTaskTemplates (PHP)
use ElementO\SecurityTasks\Domain\BrandAwareTaskTemplates;
use ElementO\SecurityTasks\Application\SecurityTaskGenerator;

// Brand name only — all tasks get "[ACME Bank]" prefix
$templates = new BrandAwareTaskTemplates('ACME Bank');

// Add a bespoke title for one specific attack
$templates = $templates->withOverride(
    'Phishing Email Campaign',
    'Conduct BEC awareness session for finance department',
);

$generator = new SecurityTaskGenerator($attackRepo, $service, $templates);

End-to-end example

From attack definition to calendar entry

1

Attack defined in DSL

A .attack file in models/ declares the attack name and category.

attack SqlInjection {
    category  APPLICATION_SECURITY
    severity  CRITICAL
}
2

Category tag resolved

SecurityTaskGenerator::categoryTag() maps APPLICATION_SECURITY → [APP].

// Internal mapping (conceptual)
AttackCategory::APPLICATION_SECURITY  →  "[APP]"
3

Brand prefix applied

The optional BrandAwareTaskTemplates prepends the brand name.

"[Globex Inc] [APP] Run SQL injection test for SqlInjection"
4

Scheduled as ProcessableItem

The task title string is handed to the scheduler, which assigns it a timestamp and persists it to the tenant's SQLite database.

// Resulting row in processable_items.sqlite:
// name:          "[Globex Inc] [APP] Run SQL injection test for SqlInjection"
// user_id:       2    (Charlie, round-robin slot)
// scheduled_at:  2026-03-09 14:20
// status:        pending

CLI reference

Running the generator

Single-tenant (default)
php bin/generate-security-tasks.php
Multi-tenant
php bin/generate-security-tasks.php --tenant=acme-corp
The script seeds three demo users (Alice, Bob, Charlie) if the users table is empty, then runs all three strategies and prints a formatted table to stdout. Processed items are written to data/processable_items.sqlite or data/{tenant}.sqlite.

Module 4 · Multi-Tenancy

Multi-Tenant Architecture

One codebase, unlimited customers. Each tenant gets an isolated SQLite database identified by a validated slug. No shared tables, no JOIN-based leaks, no extra infrastructure — just a file per customer in your $DATA_DIR.

Security & compliance

Security & data isolation

GitHub contains code only. Tenant data, SQLite files, and all runtime state live in $DATA_DIR on your infrastructure — never committed to any repository.
No secrets in the repo
Zero API keys, credentials, or tenant data are committed. Configure the DATA_DIR environment variable on your VPS. The default data/ directory is listed in .gitignore.
DB-file isolation
Each tenant slug maps to exactly one SQLite file: data/{slug}.sqlite. The slug regex /^[a-z0-9][a-z0-9\-]{0,62}[a-z0-9]$/ prevents any path-traversal sequence. An attacker cannot supply ../etc/passwd as a slug.
Schema isolation
Each SQLite file has its own schema with its own processable_items and users tables. There are no shared tables, no foreign keys crossing tenant boundaries, and no multi-tenant columns or row-level security policies to misconfigure.
Validated inputs only
ConnectionFactory::forTenant() throws InvalidArgumentException before touching the filesystem if the slug fails validation. All PDO queries use prepared statements — no raw concatenated SQL.

Module 4 · Multi-Tenancy

Multi-Tenant Architecture

One codebase, unlimited customers. Each tenant gets an isolated SQLite database identified by a validated slug. No shared tables, no JOIN-based leaks, no extra infrastructure — just a file per customer in your $DATA_DIR.

Core API

ConnectionFactory::forTenant()

The static factory method takes a tenant slug and an optional base directory, validates the slug, and returns a ConnectionFactory whose PDO connection points at {baseDir}/{tenantSlug}.sqlite. The schema is applied automatically on first connection — no migration tooling needed.

Method signature (PHP)
ConnectionFactory::forTenant(
    string $tenantSlug,          // e.g. "acme-corp"
    string $baseDir = 'data',    // directory for all tenant SQLite files
): ConnectionFactory
Usage in application code
$factory = ConnectionFactory::forTenant('acme-corp', $projectRoot . '/data');
// Creates (if absent): data/acme-corp.sqlite
// Applies schema automatically on first PDO connect

Input validation

Slug validation rules

Slug validation is enforced by a strict regex in forTenant():

Validation regex (PHP)
/^[a-z0-9][a-z0-9\-]{0,62}[a-z0-9]$|^[a-z0-9]$/
RuleReason
Lowercase letters and digits only (plus hyphen)Prevents shell injection and filesystem ambiguity
Must start and end with a letter or digitPrevents leading/trailing hyphens in filenames
Maximum 64 charactersKeeps filenames within OS path-length constraints
No dots, slashes, or spacesPrevents path-traversal attacks — cannot navigate to ../
An invalid slug throws InvalidArgumentException immediately — before any filesystem access occurs.
Valid examples:
acme-corp globex-inc tenant1 my-org-2026
Invalid examples:
../etc/passwd -leading-hyphen Uppercase has spaces

Data isolation

Tenant slug → SQLite file → isolated data

  CLI / application code
         │
         ▼
  ConnectionFactory::forTenant("acme-corp", "./data")
         │  validates slug
         │  mkdir -p ./data/
         │
         ▼
  PDO("sqlite:./data/acme-corp.sqlite")
         │  applies schema once on first open
         │
         ├─► processable_items  table  (acme-corp rows only)
         └─► users              table  (acme-corp rows only)

  ConnectionFactory::forTenant("globex-inc", "./data")
         │
         ▼
  PDO("sqlite:./data/globex-inc.sqlite")
         │
         ├─► processable_items  table  (globex-inc rows only)
         └─► users              table  (globex-inc rows only)

  No cross-tenant joins possible — separate files.

CLI reference

CLI options

generate-security-tasks.php
# Single-tenant (file: data/processable_items.sqlite)
php bin/generate-security-tasks.php

# Specific tenant (file: data/acme-corp.sqlite)
php bin/generate-security-tasks.php --tenant=acme-corp

# processable-items-demo.php also supports --tenant
php bin/processable-items-demo.php --tenant=globex-inc
Both CLI scripts accept --tenant=<slug>. When the flag is absent they fall back to the shared processable_items.sqlite for local development.

Deployment path A

Hosted service

The multi-tenant design maps directly onto a SaaS model. Each customer corresponds to one slug; the operator runs nightly cron jobs that call bin/run-all-tenants.php to push a fresh task stream to every tenant simultaneously.

The script reads a tenants.json file (or a TENANTS environment variable) to discover all active slugs, then iterates and produces per-tenant summaries.

tenants.json — operator-managed slug list
["acme-corp", "globex-inc", "initech", "soylent-red"]
Nightly cron example
# /etc/cron.d/element-io
0 3 * * 1-5 www-data php /srv/element-io/bin/run-all-tenants.php \
    >> /var/log/element-io/nightly.log 2>&1
Nightly run output (per-tenant summary)
[acme-corp]    Scheduled 15 tasks for 3 users  (5 per user)
[globex-inc]   Scheduled 24 tasks for 4 users  (6 per user)
[initech]      Scheduled 10 tasks for 2 users  (5 per user)
Done. 3 tenants updated in 0.42 s.
Horizontal scaling is trivial at this data size. SQLite files can be stored on NFS or S3-backed FUSE mounts. Upgrade to PostgreSQL later by swapping ConnectionFactory — the service layer is infrastructure-agnostic.

Deployment path B

Self-hosted appliance

Customers who want full data sovereignty can run the entire stack on their own infrastructure. The only requirements are PHP 8.2 and write access to a data directory. No external services, no database server, no cloud account.

Minimal self-hosted setup
git clone https://github.com/omer3kale/ELEMENT.-O-cyber-attacks.git
cd ELEMENT.-O-cyber-attacks
composer install
cp .env.example .env        # set DATA_DIR and TZ
php bin/generate-security-tasks.php --tenant=my-org
Container hint: The CLI + SQLite backend are a natural fit for container packaging. A future Dockerfile would need only a PHP 8.2 base image, composer install, and a mounted volume at $DATA_DIR. No daemon, no port — run as a one-shot job (docker run --rm).
.env.example
DATA_DIR=./data
TZ=UTC

Security & compliance

Security & data isolation

GitHub contains code only. Tenant data, SQLite files, and all runtime state live in $DATA_DIR on your infrastructure — never committed to any repository.
No secrets in the repo
Zero API keys, credentials, or tenant data are committed. Configure the DATA_DIR environment variable on your VPS. The default data/ directory is listed in .gitignore.
DB-file isolation
Each tenant slug maps to exactly one SQLite file: data/{slug}.sqlite. The slug regex /^[a-z0-9][a-z0-9\-]{0,62}[a-z0-9]$/ prevents any path-traversal sequence. An attacker cannot supply ../etc/passwd as a slug.
Schema isolation
Each SQLite file has its own schema with its own processable_items and users tables. There are no shared tables, no foreign keys crossing tenant boundaries, and no multi-tenant columns or row-level security policies to misconfigure.
Validated inputs only
ConnectionFactory::forTenant() throws InvalidArgumentException before touching the filesystem if the slug fails validation. All PDO queries use prepared statements — no raw concatenated SQL.

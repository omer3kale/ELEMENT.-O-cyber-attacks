Module 2 · ProcessableItems

Scheduler

ProcessableItemsService distributes work items across a date range while honouring office hours, German public holidays, and per-user minimum time spacing. Three pluggable distribution strategies let you choose between predictable compliance runs and realistic red-team unpredictability.

Domain model

ProcessableItem lifecycle

A ProcessableItem is the fundamental unit of scheduled work. It carries a name (the task title string), a userId (which analyst owns it), a scheduledAt timestamp, and a status that starts at pending and transitions to processed once the analyst marks it complete.

Items are persisted to SQLite via ProcessableItemRepository. The schema is created automatically on first connection — no migration tooling required. 

  create()          mark complete
     │                    │
  pending  ──────────►  processed
     │
  stored in {tenant}.sqlite

Pluggable distribution

Three strategies, one interface

Pass a DistributionStrategy enum case to generateAndSchedule() to select how tasks are spread across the scheduling window.

⚖️
EVEN
Tasks are distributed at mathematically equal intervals across eligible slots. Fully deterministic — the same inputs always produce the same schedule. Best for compliance audits and regression testing.
🎲
RANDOM_SPACED
Slots are chosen with uniform randomness, subject to a minimum-spacing floor. Non-deterministic — yields a different sequence on every run. Best for realistic red-team exercises where predictable timing would allow adversary preparation.
📈
WEIGHTED
Earlier available slots have lower weight; later slots have progressively higher weight. Results in a late-day bias — tasks tend to cluster in the afternoon. Best for security awareness training targeting end-of-day cognitive load.
Strategy Deterministic? Distribution shape Best use case
EVEN ✅ Yes Uniform — equal gaps Compliance audits, regression suites
RANDOM_SPACED ❌ No Uniform random, spacing-floored Red-team simulations, threat emulation
WEIGHTED ❌ No Exponential late-day bias Security awareness, cognitive-load training

Algorithm overview

How EVEN and WEIGHTED work

EVEN — conceptual algorithm
eligible_slots  ← all datetime ticks in [start, end] during office hours
                  minus holidays and already-used slots
gap             ← floor(count(eligible_slots) / item_count)
for i in range(item_count):
    slot ← eligible_slots[i * gap]   // equal step
    assign(user, slot)
WEIGHTED — conceptual algorithm
eligible_slots  ← same as EVEN
weights         ← [1, 2, 3, …, n]   // index ≈ position in day, so later = heavier
chosen          ← weighted_random_sample(eligible_slots, weights, item_count)
for slot in sort(chosen):
    assign(user, slot)

Constraints

Office hours & German holiday calendar

GermanHolidayCalendar embeds the full set of federal German public holidays. The scheduler calls isEligible(DateTimeImmutable) on each candidate slot and skips anything outside 09:00–17:00 Mon–Fri on non-holiday days.

The minimum-spacing floor (default 30 minutes, configurable via $minDistanceMinutes) prevents two tasks for the same user from landing within the floor window. If the window does not contain enough eligible slots for the requested item count, the scheduler silently produces fewer items rather than violating the floor.

Planning API

estimateCapacity()

Question it answers: "How many tasks can I schedule for one user in this time window before I run out of eligible slots?"
Usage (PHP)
use ElementO\ProcessableItems\Application\ProcessableItemsService;

$start    = new DateTimeImmutable('next monday 09:00');
$end      = $start->modify('+4 days')->setTime(17, 0);

// With default 30-minute spacing floor:
$capacity = $service->estimateCapacity($start, $end);
// → e.g. 16 for a standard 5-day work week

// With tighter 20-minute floor:
$capacity = $service->estimateCapacity($start, $end, minDistanceMinutes: 20);

Call before seeding users to ensure you never request more tasks per user than the window can hold. Multiply by user count for total sprint capacity.

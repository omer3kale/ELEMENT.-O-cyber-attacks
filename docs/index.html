Architecture in 30 seconds

Three modules, one pipeline

Each module maps to a bounded context in the DDD layer; none depends on the others at the infrastructure level.

⚔️
Attack Catalog
30 attacks across 12 categories defined in a custom .attack DSL and parsed by an ANTLR-based adapter into AttackAggregate domain objects. No live exploit payloads — structural threat modelling only.
📅
Scheduler
ProcessableItemsService distributes work items across a date range with three pluggable strategies — EVEN, RANDOM_SPACED, WEIGHTED — while honouring office hours and German public holidays.
🏷️
Security Task Generator
SecurityTaskGenerator bridges the catalog and the scheduler. Each attack becomes a tagged, brand-prefixed task title like [ACME Bank] [SOCIAL-ENG] Run phishing simulation….

Example flows

How this works in practice

1

Single-tenant run

Parse the attack catalog, seed three demo users, and schedule 5 tasks per user over the next work week using all three distribution strategies. 

composer install
php bin/generate-security-tasks.php

Output is written to data/processable_items.sqlite.

2

Multi-tenant run

Provide a --tenant slug and each customer gets a fully isolated SQLite database — no shared tables, no cross-tenant risk. 

php bin/generate-security-tasks.php --tenant=acme-corp
php bin/generate-security-tasks.php --tenant=globex-inc

Creates data/acme-corp.sqlite and data/globex-inc.sqlite independently.

3

Capacity planning with estimateCapacity()

Before committing to a schedule, estimate how many items can fit in a time window given a minimum spacing floor. 

// In your own PHP script or REPL:
$start    = new DateTimeImmutable('next monday 09:00');
$end      = $start->modify('+4 days')->setTime(17, 0);
$capacity = $service->estimateCapacity($start, $end, minDistanceMinutes: 30);
echo "Slots available: {$capacity}\n";

Useful for sizing a sprint before seeding the database.

Safety-first design

No secrets. No live payloads. No cross-tenant risk.

GitHub contains code only. Tenant data, SQLite files, and all runtime state live in $DATA_DIR on your infrastructure — never committed to any repository.
No tokens in the repo
No API keys, credentials, or tenant data are stored in version control. Configure secrets as environment variables on your VPS.
DB-file isolation
Each tenant slug maps to exactly one SQLite file. The slug regex prevents path traversal. No shared tables, no JOIN-based leaks.
Structural modeling only
Attack definitions describe threat categories and scheduling context — not live exploit payloads or proof-of-concept code.
